# VISION —— 骑行全周期智能助手(产品愿景)

> 当前仓库是这个愿景的第一阶段:摩托车售后智能客服 Agent。
> 本文档记录完整产品形态与实施顺序,防止愿景丢失;实施严格按阶段推进,不干扰当前主线。

## 产品定位

从"店家的售后客服"进化为"骑士的全周期助手",覆盖四个场景:

1. **修车/保养**:查攻略、查配件兼容型号
2. **日常骑行**:兜风地点推荐、天气汇报、辅助切换导航
3. **紧急救援**:快速给出靠谱修车店(位置、电话、拖车服务、价格),故障预测 + 维修价格估算
4. **买卖二手车**:车价评估

## 功能地图与架构映射

| 功能 | 现有底盘 | 还缺什么 | 难点性质 |
|---|---|---|---|
| 攻略 + 配件型号 | ✅ HybridRetriever(RAG)+ Neo4j 图谱 | 语料扩充到更多车型 | 数据搬运 |
| 兜风推荐 + 天气 + 导航切换 | supervisor 路由 + ActionAgent 工具模式 | 天气 API / POI API / 导航 deep-link 三个工具,经 MCP 接入 | 纯工具集成,最容易 |
| 紧急救援修车店 | 工具模式 + 路由 | 商家数据库(核心是"靠谱"数据的来源:人工筛选 + 用户评价)、地理位置查询 | 数据运营 > 技术 |
| 故障预测 + 维修估价 | diagnosis 路由(拆解 + HyDE)已是雏形 | 故障案例库、工时费/配件价数据 | 准确性与责任:输出必须是区间 + 免责 + 人工兜底,上线前必须有评估集 |
| 二手车估价 | — | 成交价数据集(最大瓶颈) | 有数据是回归问题,没数据就是拍脑袋;最后做 |

## 阶段规划

- **一期(当前)**:售后 Agent 核心能力 + 评估体系 + 工程化
- **二期**:React 前端 + Spring Boot 业务系统 + FastAPI Agent 服务的三层架构
- **三期**:按难度序逐个上——工具集成(兜风/天气/导航)→ 数据运营(救援商家库)→ 数据科学(估价)

## 设计原则(从一期继承)

- 新功能 = 新工具 + 新路由,不推翻底盘(supervisor 模式的扩展性)
- 涉及钱和安全的输出(维修报价、救援推荐)必须有护栏:估算区间、免责声明、人工审核兜底
- 每个功能上线前先造评估集,数字说话
