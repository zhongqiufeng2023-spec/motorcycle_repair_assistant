## 二期工单化方案(整体架构,2026-07-19 讲解定稿)

**核心思路:审批从"对话的暂停键(interrupt)"→"独立工单(refund_tickets 表 + 状态机)"。** 对话线程发起工单后立刻正常结束,不挂起;商家在业务系统批复;结果异步推回对话。

**一笔退款的旅程:**
1. 用户"要退货"→ action_node 的 LLM 决定调 request_refund → 工具不再直接退款,而是**调业务系统开工单**(status=PENDING)→ 对话立刻回"已提交单号T001,审核后通知"→ **这句进账本(有始有终)**→ 线程干净结束。
2. 用户继续插话 → 账本里退货已有下文 → LLM 不复活它 → 僵尸死。
3. 商家控制台看 PENDING 工单 → 通过/驳回 → 更新工单状态 → 通过则**工单系统确定性执行退款(无 LLM)**→ 结果推回对话。

**四坑一次清(同一病根=审批寄生对话线程):** 僵尸复活 / 商家 yes 蒸发 / 审批锁会话 / /approve 裸奔。

**三层架构:**
- React:用户聊天窗 + 商家控制台(待审工单列表 + 通过/驳回)
- FastAPI(Agent 层):对话/路由/检索;request_refund 工具改为"调业务系统开工单"(接口契约不变,mock→真实,action_node 几乎不动)
- Spring Boot(业务系统):refund_tickets 表 + 状态机 + 鉴权 + 执行退款 + 通知(业务规则唯一权威)

**推送方式(二期开工时定):** 轮询(最简,客服场景够用)vs WebSocket/SSE(实时,多维护长连接)。待定。

**范围锁死:** 只把现有退款审批工单化跑通,不铺 VISION 新功能(那是三期)。目标:一期二期下周内完成可实用客服。

**interrupt 的定位认知(面试素材):** interrupt 适合"单事务的同步中途卡点"(D8 单测完美);不适合"长时间可打断、与对话并行的审批"——后者正确工具是工单+状态机+异步通知。知道工具适用边界 > 会用工具。

## 二期记忆架构蓝图(2026-07-20 讨论定,详见 REVIEW.md 9.5)

**"记忆"拆四层,各住各地——别混成一个 DB:**
| 层 | 是什么 | 现在 | 二期存哪 | 键 |
|---|---|---|---|---|
| ①会话内工作记忆 | 当前对话 messages | MemorySaver(内存,重启丢) | 持久 checkpointer:Postgres/Redis | thread_id |
| ②跨会话长期记忆 | 用户车型/偏好/历史订单/过往摘要 | 无 | 结构化→Spring Boot 业务库;语义→Chroma(按 user_id 过滤) | user_id |
| ③领域知识 | 手册/配件 | Chroma+Neo4j | 维持 | 全局 |
| ④长上下文 | 对话塞不进窗口 | 滑窗 n=10 | 加摘要压缩/对历史检索 | — |

**两把刀(别混):** DB 治"存不下/重启丢";context engineering(滑窗/摘要/对历史检索)治"喂给 LLM 太长"。DB 存百万条也喂不进 LLM——存全量、喂精选,两件独立的事。逼你上 DB 的是"多用户+不丢",不是单段对话变长。

**LangGraph 原生分工:** Checkpointer=短期(线程内,thread_id)/ Store(langgraph.store)=长期(跨线程,user_id 命名空间,部分支持语义检索)。②这层二期用 Store 或业务库实现。

**落地顺序(YAGNI,现在别做):** ①换持久 checkpointer(和工单化同批,重启不丢挂起审批)→ ②用户系统就绪后加跨会话记忆 → ④对话真长了再上摘要压缩。现在 MemorySaver+n=10 够用。
