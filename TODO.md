## D5(图检索)期间推迟的

- [ ] **Pydantic 结构化校验**
  - 是什么:用 Pydantic 卡住 LLM 的结构化输出(字段、类型、默认值)。
  - 为什么推迟:当前 Text2Cypher 里 LLM 只吐一条 Cypher 字符串,结构极简,套 Pydantic 是仪式感大于收益(过度工程)。两层护栏(只读强制 + EXPLAIN 预校验)已够挡危险查询。
  - 什么时候回来:D6 需要从用户问题里**结构化抽取多字段**(如 `{品牌, 车型, 年份}`)时,Pydantic 才名副其实——那时再上。

- [ ] **图问题的拆解 / 路由**
  - 是什么:判断一个问题该走图还是走向量;复合图问题拆成子问题。
  - 为什么推迟:这是 D6「双引擎路由」的职责,不该塞进 D5 的检索器(检索器只管检索,不判断该不该用自己)。
  - 什么时候回来:D6。且**复用 D4 已有的 `decompose_query`**,不要重写。

- [ ] **schema 自动导出**
  - 是什么:用 `CALL db.schema.visualization()` 自动生成图谱 schema 字符串,替代手写。
  - 为什么推迟:当前图小且固定,手写 schema 更清晰,还能顺手把业务规则(如年份区间判断)写进去,自动导出给不了这个。
  - 什么时候回来:节点/关系类型变多、手维护 schema 字符串变累时。

- [ ] **只读的数据库级兜底(Neo4j 只读权限用户)**
  - 是什么:给数据库开一个只有读权限的账号,让数据库本身拒绝写操作,而不只靠应用层关键词扫描。
  - 为什么推迟:应用层 `WRITE_PATTERN` 正则扫描当前已够用。
  - 什么时候回来:项目工程化 / 部署(D9 或 D12)做纵深防御时。
